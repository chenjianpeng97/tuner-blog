# 测试部门管理 v1

## 背景

目前随着俺要求给测试部门加 HC 的要求的实现，测试人员已经从我入职时候的 2 个正式员工发展到 4 个正式员工+1 实习+1 外包的模式了
理论上测试工程师 over whelm by chore work 的现象应该有些改善，那么作为 QA leader 的俺就应该准备~~给领导画饼~~筹备一下测试部门下一阶段的建设计划了，让老板觉得这钱花的值哇

## 现状

- 工作分配

有些项目以前安排的不太合理，有些交付项目的几个子系统测试负责人不同，可能导致一些子系统集成功能测试过程中出现遗漏
入职时间较长的员工承担了较多的老项目维护工作，有时这些项目的甲方业务迭代导致需求 backlog 大量堆积，其他人又缺少业务知识和项目经验难以完全主导测试

- 人员情况

今年是测试比较难熬的一年，在没有自动化测试套件的情况下重构了两个大型项目，手工测试时依赖新旧系统功能行为对比，工作量是普通项目 1.5~2 倍左右
此外还有一些测试工作之外的投入，如外包甲方的其他系统 UAT 验收测试，质量管理，项目考核等等
从缺陷数量来定义工作量的角度，今年新建的 bug 2822 个比去年 1586 个增长 77.9%；加上增加的用例评审活动，质量考核活动，外包测试工作等产出应可以算增长了 100% 左右

T 老师在职时间最长，对公司和甲方业务接触最多，工作稍带情绪化但责任心强；课题上适合安排测试资源管理类；能否能带好初级工程师完成 B 类任务的能力有待观察，可能需要帮助
Y 老师细心负责，沟通能力强，学习速度快，能把负责的系统模块功能反述的十分清晰有条理；课题上适合安排测试资源管理类，工具平台类
Q 老师刚入职，面试表现和入职沟通感觉测试经验没问题，也有技术实力，其他特征还需进一步观察；课题上准备安排测试技术类
W 同学是实习生，初入职场各种能力比较小白，但责任心可以，理论知识比较扎实，但结合工作实际还需磨练，适合手把手培养，争取转正做初级工程师

- 测试团队能力

自动化测试还在实验阶段
非功能需求测试能力(安全，性能，静态代码检查)在入门阶段刚够交付一些 DAST 的基本扫描测试报告
用例管理已进步到应写尽写，但格式统一，定期整理，总结成用例库上还有很大可改进空间

## 改进

- 工作分配

建立以交付客户为单位的测试负责人机制，一套系统由一个中级测试工程师统一负责，管理多个子系统的需求变更与测试，作为 A 类工作；
其他测试工程师可以支援测试负责人的工作，但作为 B 类工作，由系统测试负责人负责统筹安排
以 A，B 类任务的模型统筹测试团队内正式员工的工作分配，A 类任务工作量尽量均匀;B 类任务由初级测试工程师，外包，实习，支援等方式动态调配人力问题

落地步骤:

1. 整理项目-子系统-负责人列表
2. 统筹任务分配，根据统筹情况逐步安排交接工作
3. 根据后续变化动态调整(甲方终止合作,增加子系统造成的动态变化)

- 测试团队能力建设

如果说设计测试是 A 类工作，执行测试是 B 类工作，课题研究可以列为 C 类工作；
一般 A，B 类工作是有交付周期的项目，在忙季应作为主要投入，这些工作往往涉及工作产出物的向下交付，如果有耽误的话损失是较大的；
C 类工作为研究型工作，产出的东西给团队内部使用，很难量化验收标准(如论文好坏的指标是交付后多年之后展现的'引用次数',这并不是能够在发布前被量化验收预测的)，在高压下做出来的 C 类工作成果往往难以落地，在团队失去对其的信心后被放弃，或者被管理层强制推进，全公司的人去强行用一些难用的工具或者流程，白白浪费效率；
故屯长俺认为，C 类工作最好是执行者有着热情，充分经验，足够小的交付压力下进行的；以屯长俺接触的一些开源测试工具开发者的状态，好的工具(MangoTestPlatform,xmind2testcase)时往往就是公司业务比较清闲，自身有精进事业的动力的情况下做出来的
一些大的团队会有测试开发工程师这个岗位来专门做 C 类工作，对于现在小团队俺更倾向于俺老东家某学历厂的做法--课题制
发布一些课题给团队成员,在闲季主动去完成这些课题，加以公司层面的经济奖励，来推动 C 类工作的进行

课题列表如下

- 工具平台类

  - 测试用例管理与收集(统一不同情况下用的需求追踪矩阵,xmind,禅道,excel 中的用例)
  - CICD 平台搭建，建立自动化的准入准出机制

- 测试资源管理

  - 常用功能用例库(系统测试用例)
  - 常用业务验收用例库(验收测试用例)

- 测试技术类

  - 分层的自动化测试(单元，集成，系统，验收)
  - 非功能需求测试解决方案(性能，安全)

落地步骤:

1. 公布课题给测试团队内部
2. 同项目管理，定义课题的需求到交付验收机制
3. 与管理层沟通，申请建立奖励机制
4. 落地后观察效果，持续改进
